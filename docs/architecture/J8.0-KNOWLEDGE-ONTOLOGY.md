## Overview

This document is the primary deliverable for J8.0.

J7 completed the reasoning layer. The system now produces explicit, traceable strategic advice grounded in evidence.

J8 addresses the layer beneath it: how knowledge is acquired, stored, and retrieved.

Today every research run reconstructs knowledge from scratch. Documents are loaded, chunked, and mined for evidence on demand. As the corpus grows this becomes the binding constraint on system performance.

J8 introduces a **persistent knowledge platform**. Knowledge is constructed once and consumed many times. Reasoning consumes stored knowledge rather than extracting it live.

This document defines:

- what knowledge *is* in this system
- how it is represented
- how it persists
- how it is retrieved
- how the transition from J7 to J8 is managed safely

## Part 1 ‚Äî Knowledge Ontology

### 1.1 Core Principle

The canonical unit of reasoning is **Evidence**.

Evidence is an atomic, source-backed claim.

Evidence exists because one or more **Sources** support it.

Sources and Evidence are the two immutable objects in the knowledge layer. Everything else is mutable metadata about them.

### 1.3 Canonical Objects

#### Source

Represents an original document exactly as obtained from the world.

A Source never changes.

| Field | Type | Description |
|-------|------|-------------|
| `source_id` | `str` | Content-addressed ID (SHA-256 of canonical text) |
| `uri` | `str` | Original location (file path, URL, or DOI) |
| `title` | `str` | Document title |
| `author` | `str \| None` | Primary author(s) |
| `publisher` | `str \| None` | Publishing organisation |
| `publication_date` | `date \| None` | Date of publication |
| `retrieved_date` | `date` | Date this copy was obtained |
| `fingerprint` | `str` | SHA-256 of canonical text |
| `language` | `str` | ISO 639-1 language code |
| `document_type` | `SourceType` | PDF, HTML, TXT, DOCX, API |
| `domain` | `str` | Domain tag (smr, ai_data_centers, transmission) |
| `copyright` | `str \| None` | Copyright statement if extractable |
| `canonical_text` | `str` | Full extracted text (stored once, never re-extracted) |
| `page_count` | `int \| None` | Page count if known |

`source_id` is derived from `fingerprint`. Two files with identical content share a single Source record. This is the deduplication mechanism for sources.

**Invariant:** Once written, a Source record is never modified. The `canonical_text` field is populated exactly once at ingestion.

#### Evidence

Represents one atomic claim extracted from one or more Sources.

An Evidence record is **append-only**: it is never modified after creation. If a claim is superseded by improved extraction or new information, a new Evidence record is created referencing the old one via `superseded_by`.

| Field | Type | Description |
|-------|------|-------------|
| `evidence_id` | `str` | Unique ID (UUID v4) |
| `statement` | `str` | The atomic claim, self-contained |
| `supporting_source_ids` | `list[str]` | One or more Source IDs that support this claim |
| `profile_ids` | `list[str]` | Profiles this evidence is tagged to |
| `extraction_run_id` | `str` | Which ExtractionRun produced this |
| `entity` | `str` | Named entity in the claim (e.g. "GB200 NVL72") |
| `entity_type` | `str` | Entity category (e.g. "rack", "reactor_unit") |
| `scope` | `str` | Measurement scope (e.g. "fleet", "component") |
| `category` | `EvidenceCategory` | Technical / Economic / Regulatory / etc. |
| `supersedes` | `list[str]` | Evidence IDs this replaces |
| `superseded_by` | `str \| None` | If not None, this record is retired |
| `contradiction_ids` | `list[str]` | Contradiction records involving this evidence |

**What Evidence is NOT:**

Evidence does not contain:
- chunk references (Chunk is a builder artifact, not a knowledge object)
- mutable quality scores (these live in KnowledgeMetadata)
- embeddings (these live in the embedding index, referenced by evidence_id)
- raw snippets (the full Source is the provenance; snippets are extraction artifacts)

The statement field must be self-contained. A reader must be able to understand it without reading the source.

#### KnowledgeMetadata

Mutable quality and lifecycle metadata for an Evidence record.

This is separated from Evidence deliberately. Evidence records must remain immutable for provenance purposes. Quality assessments evolve as the knowledge base matures.

| Field | Type | Description |
|-------|------|-------------|
| `evidence_id` | `str` | Foreign key to Evidence |
| `state` | `EvidenceState` | ACTIVE, SUPERSEDED, LOW_CONFIDENCE, RETRACTED, ARCHIVED |
| `confidence` | `float` | 0.0‚Äì1.0 |
| `credibility` | `CredibilityLevel` | HIGH, MEDIUM, LOW |
| `relevance_score` | `float` | 1.0‚Äì5.0 |
| `source_quality_score` | `float` | 1.0‚Äì5.0 |
| `specificity_score` | `float` | 1.0‚Äì5.0 |
| `overall_score` | `float` | Computed composite |
| `review_status` | `ReviewStatus` | UNREVIEWED, AUTO_REVIEWED, HUMAN_REVIEWED |
| `version` | `int` | Monotonic version counter |
| `created_at` | `datetime` | |
| `updated_at` | `datetime` | |
| `quality_flags` | `list[str]` | e.g. ["synthetic_source", "low_specificity"] |

EvidenceState values: `ACTIVE`, `SUPERSEDED`, `LOW_CONFIDENCE`, `RETRACTED`, `ARCHIVED`

**Design rule:** If quality scoring logic changes, update KnowledgeMetadata records. Never touch Evidence records.

#### Contradiction

A first-class object representing a known conflict between two Evidence records.

Contradictions are computed once and stored permanently. They are not regenerated on each research run.

| Field | Type | Description |
|-------|------|-------------|
| `contradiction_id` | `str` | UUID v4 |
| `evidence_id_a` | `str` | First evidence |
| `evidence_id_b` | `str` | Conflicting evidence |
| `contradiction_type` | `ContradictionType` | DIRECT, SCOPE, TEMPORAL, METHODOLOGICAL |
| `detection_method` | `DetectionMethod` | AUTOMATED, HUMAN |
| `severity` | `ContradictionSeverity` | HIGH, MEDIUM, LOW |
| `resolution_status` | `ResolutionStatus` | OPEN, RESOLVED, ACCEPTED_AMBIGUITY |
| `resolution_notes` | `str \| None` | Explanation when resolved |
| `detected_at` | `datetime` | |

The current system detects contradictions at runtime during each research run. Under J8, this becomes a background KB process. Research runs consume pre-computed contradiction relationships.

#### ExtractionRun

Records the parameters and outcome of a Knowledge Builder execution.

This is the mechanism for managing evidence versioning when extraction quality improves (e.g. new model, improved prompt).

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | `str` | UUID v4 |
| `source_ids` | `list[str]` | Sources processed in this run |
| `model_version` | `str` | LLM model identifier |
| `prompt_version` | `str` | Extraction prompt version tag |
| `evidence_ids_produced` | `list[str]` | Evidence created |
| `evidence_ids_superseded` | `list[str]` | Evidence retired |
| `started_at` | `datetime` | |
| `completed_at` | `datetime` | |
| `status` | `RunStatus` | RUNNING, COMPLETED, FAILED |

ExtractionRun is the mechanism by which the KB evolves without corrupting immutable Evidence records. A new extraction run produces new Evidence records and marks old ones as `SUPERSEDED`. The old records remain queryable for audit purposes.

## Part 3 ‚Äî Object Ownership Rules

| Object | Owner | Mutable? | Rule |
|--------|-------|----------|------|
| Source | Knowledge Builder | No | Written once at ingestion; fingerprint enforces uniqueness |
| Evidence | Knowledge Builder | No | Append-only; superseded records are retired, never modified |
| KnowledgeMetadata | Knowledge Base | Yes | Scores and states evolve; version counter enforces ordering |
| Contradiction | Knowledge Base | Partially | Detection is immutable; resolution_status and notes are mutable |
| ExtractionRun | Knowledge Builder | Partially | Status field updates during run; rest is immutable after completion |
| EvidenceProfile | Knowledge Base | Yes | Profile tags can be added and removed |
| Embedding | Embedding Index | No | Tied to evidence_id; replaced only when evidence is superseded |
| Decision Graph | Orchestrator | No | Produced by each research run; archived after run completes |
| Executive Report | Report Agent | No | Presentation artefact; not a KB object |

**The fundamental rule:** objects owned by the Knowledge Builder are never mutated by the research pipeline. The research pipeline is a consumer, not a writer.

## Part 4 ‚Äî Persistence Strategy

### 4.1 Storage Tier Recommendation

For the expected corpus scale (1K‚Äì100K documents over five years), two tiers are sufficient:

**Tier 1 ‚Äî Structured Records**  
JSON Lines (`.jsonl`) files with one object per line, organized by domain and type. Simple, human-readable, Git-compatible, zero infrastructure dependency.

**Tier 2 ‚Äî Embedding Index**  
A local vector database. Recommended: **Qdrant** (local mode). Reasons: runs embedded with no server, stores payload alongside vectors, supports filtering by profile/domain/state, has a clean Python API, and scales to 10M+ vectors on local hardware.

### 4.2 File-Based Record Layout

Structured records are stored as JSONL with a companion index:

```
knowledge/sources/{domain}/{source_id}.json
knowledge/evidence/{domain}/evidence.jsonl
knowledge/evidence/{domain}/index.json          ‚Üê lookup table: evidence_id ‚Üí file offset
knowledge/metadata/{domain}/metadata.jsonl
knowledge/contradictions/contradictions.jsonl
knowledge/extractions/runs.jsonl
```

Each domain directory is self-contained. Cross-domain evidence is stored in `knowledge/evidence/cross-domain/`.

### 4.3 Atomicity

Writes follow a write-then-rename pattern:

1. Write new records to `{filename}.tmp`
2. Append to index
3. Rename `.tmp` to final path

This prevents partial writes from corrupting the KB. No transaction manager is required at current scale.

## Part 5 ‚Äî Versioning Strategy

### 5.1 Source Versioning

Sources are content-addressed by `fingerprint` (SHA-256 of canonical text). Two files with identical content are one Source. A revised document produces a new Source with a new `source_id`. The old Source remains intact.

### 5.2 Evidence Versioning

Evidence is versioned through the `supersedes` / `superseded_by` relationship:

```
Evidence A (extraction_run=run_001)  [state=SUPERSEDED]
    ‚îÇ
    ‚îî‚îÄ superseded_by: Evidence B

Evidence B (extraction_run=run_002)  [state=ACTIVE]
    ‚îÇ
    ‚îî‚îÄ supersedes: [Evidence A]
```

When an ExtractionRun produces improved evidence for a source, it:
1. Creates new Evidence records
2. Sets `supersedes` on new records
3. Sets `superseded_by` on old records
4. Updates KnowledgeMetadata state to `SUPERSEDED` on old records

Old evidence records remain in the KB for audit, comparison, and rollback.

### 5.3 Schema Versioning

Each JSONL file carries a `schema_version` field in a header record (line 0). Schema migrations are additive (new optional fields only) until a breaking change requires a migration script. Breaking schema changes increment the major version and require a migration that reads old format and writes new format.

## Part 6 ‚Äî Retrieval Strategy

### 6.1 Current vs Target Architecture

**Current (J7):**
```
Question ‚Üí Load all sources ‚Üí Chunk ‚Üí Score chunks ‚Üí Extract evidence ‚Üí Reason
```

Every query touches the filesystem and runs an LLM over raw text.

**Target (J8):**
```
Question ‚Üí Retrieve evidence from KB ‚Üí Reason over retrieved evidence
```

Documents are never loaded during a research run unless explicitly requested.

### 6.2 Evidence Retrieval Pipeline

```
Planner produces queries
        ‚Üì
EvidenceRetriever:
  - embed each query
  - vector search over evidence embeddings
  - filter by: profile, domain, state=ACTIVE
  - return top-k evidence items with KnowledgeMetadata
        ‚Üì
Evidence set passed to Decision Graph
        ‚Üì
[Optional] Lazy source loading:
  If an agent requires the full source text for a specific evidence_id,
  it loads Source.canonical_text on demand.
```

### 6.3 Retrieval Filters

The EvidenceRetriever supports compound filters:

- `profile_id` ‚Äî restrict to evidence tagged for a specific profile
- `domain` ‚Äî restrict by domain (smr, ai_data_centers, transmission)
- `state` ‚Äî default filter: `ACTIVE` only (exclude SUPERSEDED, RETRACTED)
- `category` ‚Äî filter by EvidenceCategory
- `min_overall_score` ‚Äî quality threshold
- `entity` ‚Äî filter by named entity

Filters are applied at the vector DB layer for efficiency, not in post-processing.

### 6.4 Fallback to Live Extraction

During the migration period (J8.1‚ÄìJ8.3), live extraction remains available as a fallback when the KB does not contain evidence for a requested domain or source. The EvidenceRetriever signals `insufficient_kb_coverage` and the Planner falls back to the J7 live pipeline.

## Part 7 ‚Äî Embedding Analysis: Option C Evaluation

The proposal offers three options for embedding granularity. This section evaluates them quantitatively.

### 7.1 Scale Assumptions

| Parameter | Value | Basis |
|-----------|-------|-------|
| Avg. pages per document | 30 | Observed corpus (PDFs range 5‚Äì120 pages) |
| Evidence items per document | 15‚Äì20 | Conservative extraction yield after dedup |
| Chunks per document | 50‚Äì75 | At 500-token chunk size |
| Embedding dimensions | 1,536 | `text-embedding-3-small` (default recommendation) |
| Bytes per float | 4 | float32 |
| Bytes per embedding | 6,144 | 1,536 √ó 4 |
| Index overhead factor | 2√ó | HNSW graph overhead |

### 7.2 Scale Projections

**1,000 documents**

| Approach | Count | Raw storage | With index |
|----------|-------|-------------|------------|
| Option A: Source-level | 1,000 | 6 MB | 12 MB |
| Option B: Chunk-level | 62,500 | 384 MB | 768 MB |
| Option C: Evidence-level | 17,500 | 108 MB | 215 MB |

**10,000 documents**

| Approach | Count | Raw storage | With index |
|----------|-------|-------------|------------|
| Option A: Source-level | 10,000 | 62 MB | 123 MB |
| Option B: Chunk-level | 625,000 | 3.8 GB | 7.6 GB |
| Option C: Evidence-level | 175,000 | 1.1 GB | 2.1 GB |

**100,000 documents**

| Approach | Count | Raw storage | With index |
|----------|-------|-------------|------------|
| Option A: Source-level | 100,000 | 614 MB | 1.2 GB |
| Option B: Chunk-level | 6,250,000 | 38 GB | 76 GB |
| Option C: Evidence-level | 1,750,000 | 10.7 GB | 21 GB |

After cross-document evidence deduplication (estimated 20‚Äì30% reduction), evidence-level storage is approximately **10‚Äì15 GB** at 100,000 documents.

### 7.3 Verdict

**Option C is recommended. The proposed concern about larger index size is wrong.**

Evidence-level embeddings are 3‚Äì4√ó *smaller* than chunk-level embeddings at every scale, not larger. Chunks are a mechanical subdivision of documents. Evidence is the distilled semantic content. The extraction process acts as a compression step.

| Dimension | Option A (Source) | Option B (Chunk) | Option C (Evidence) |
|-----------|------------------|-----------------|-------------------|
| Index size | Smallest | Largest | Middle (3‚Äì4√ó smaller than chunks) |
| Retrieval precision | Coarse | Medium | High |
| Deduplication | Possible | Difficult | Natural |
| Semantic alignment | Weak | Incidental | Direct |
| Practical at 100K docs | Yes | Marginal | Yes |

At 100,000 documents, 21 GB fits comfortably in a single Qdrant instance on modern hardware. This is not a constraint that requires distributed infrastructure.

**The "larger index" concern in the proposal should be retired. Evidence-level embeddings are strictly preferred at all scales.**

## Part 10 ‚Äî Risks and Tradeoffs

### 10.1 Evidence Extraction Quality

**Risk:** Evidence quality depends on the LLM and prompt version at extraction time. Evidence extracted today may be inferior to evidence extracted in 12 months with an improved model.

**Mitigation:** ExtractionRun versioning and the supersedes pattern explicitly support re-extraction. The KB is designed to evolve without losing history.

**Residual risk:** Low. The architecture was designed with this problem in mind.

---

### 10.2 Corpus Coverage Gaps

**Risk:** The KB covers only what has been explicitly ingested. A research run on a new domain or question will miss evidence from sources not yet in the KB.

**Mitigation:** The fallback to live extraction (Phase 5) handles this. Coverage checks in the EvidenceRetriever signal when KB coverage is insufficient.

**Residual risk:** Medium during migration; Low once KB is mature.

---

### 10.3 Evidence Granularity

**Risk:** Atomic evidence statements may lose context that was present in the original chunk or page. A claim that reads correctly in isolation may be misleading without surrounding context.

**Mitigation:** Each Evidence record carries `supporting_source_ids`. Lazy loading of `Source.canonical_text` is available to any agent that needs fuller context. The statement field specification requires self-contained claims, which disciplines extraction quality.

**Residual risk:** Medium. Requires discipline in extraction prompts.

---

### 10.4 Immutability vs Correction

**Risk:** If an Evidence record contains an error (wrong claim, wrong source attribution), immutability prevents direct correction.

**Mitigation:** The `RETRACTED` state in KnowledgeMetadata marks an Evidence record as invalid without deleting it. A corrected replacement is created as a new Evidence record with a `supersedes` reference.

**Residual risk:** Low. The pattern is well-defined.

---

### 10.5 Embedding Index Staleness

**Risk:** If new Evidence is added without being embedded, the vector index is out of sync with the JSONL store.

**Mitigation:** `manifest.json` in the embedding directory tracks which evidence_ids have been indexed. The KnowledgeBuilder checks for unindexed evidence at startup and embeds it before returning. The `stats.json` file in `_meta/` exposes coverage metrics.

**Residual risk:** Low. Detectable and recoverable.

---

### 10.6 Single-Node Vector DB Constraint

**Risk:** At 100,000 documents, the Qdrant index is ~21 GB. If the corpus scales beyond 100,000 documents, a single local Qdrant instance may require a distributed architecture.

**Assessment:** 100,000 documents represents approximately 5‚Äì10 years of corpus growth at current intake rates. The single-node constraint is not a near-term risk.

**Mitigation:** Qdrant offers a distributed mode and a managed cloud option. The EvidenceRetriever interface abstracts the vector DB implementation. Swapping to distributed Qdrant or another provider requires no changes to the research pipeline.

**Residual risk:** Low within the J8 planning horizon.

## Recommended Architecture

### Summary Recommendation

The proposed ontology is sound. The recommendations below refine rather than replace it.

**Adopt:**

1. `Source ‚Üí Evidence ‚Üí Decision Graph` as the canonical three-layer stack
2. Source immutability (content-addressed by fingerprint)
3. Evidence append-only semantics (supersedes pattern, not in-place modification)
4. **Separate KnowledgeMetadata from Evidence.** This is the most important refinement. Putting mutable quality scores directly on an immutable Evidence object creates an internal contradiction. KnowledgeMetadata is the correct owner of lifecycle and quality state.
5. ExtractionRun as the versioning mechanism for evidence evolution
6. Contradiction as a first-class KB object (not a per-run computation)
7. Option C ‚Äî evidence-level embeddings (the "larger index" concern is incorrect; see Part 7)
8. Chunk as a builder artifact only ‚Äî never a KB object
9. Qdrant local mode for the embedding index
10. Staged migration through J8.1‚ÄìJ8.6

**Retire:**

- The concern that evidence-level embeddings produce a larger index than chunk-level (they do not)
- The idea of raw snippet storage on Evidence (the Source is the provenance; snippets are extraction artifacts)

**Defer to J9:**

- Distributed vector DB infrastructure
- Persistent web crawl and incremental web source ingestion
- Enterprise workspace isolation
- Knowledge base API (HTTP) ‚Äî the KB is file-backed through J8

### Design Statement

> The Knowledge Base is the memory of the system.  
> The Decision Graph is its reasoning.  
> These are different things, owned by different processes, with different lifecycle rules.  
> J8 establishes that boundary permanently.
